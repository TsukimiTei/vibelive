# Vibelive

一句话描述你的产品：

 我需要做一个只focus为we code正在co定项目啊，就直播AI写代码的直播平台，然后需要非常有个性，然后要游戏化一些。可以有一些差异化的功能设计，比如直播中会智能隐藏 api key，或者 url 的泄露。可以自动识别主播正在使用的 coding工具。然后观众不光是点赞，可以点想用，有趣，期待等。你可以再想想其他的。

**Q: Vibelive 的核心定位是什么？请选择最符合你想法的方向：**

A: 纯直播平台：用户观看 AI 实时写代码的过程（类似 Twitch 看别人写代码）

**Q: 目标用户主要是哪类人群？**

A: 以上都要（包括：开发者/程序员：想学习或围观 AI 辅助编程（Vibe Coding）的技术人员、AI 爱好者/非技术用户：对 AI 写代码感到好奇的普通观众、内容创作者/主播：想通过直播 AI 编码过程来吸引粉丝的博主）

<br />

**Q: 技术实现方面，你倾向于哪种架构？**

A: 先做 Web 应用 MVP，后续再扩展到移动端

**Q: 「直播 AI 写代码」的内容来源是什么？**

A: 主播在本地使用 AI 编程工具（如 Cursor、Copilot），平台捕获屏幕/终端进行直播

<br />

**Q: 直播流的技术方案你倾向于哪种？这决定了主播端的推流方式和观众端的延迟体验：**

A: 使用第三方直播服务（如 Mux、Agora、LiveKit）快速搭建，降低自研成本

**Q: 关于「智能隐藏 API Key / URL」等安全功能，你期望的实现层级是？**

A: 客户端 + 服务端双重检测，确保万无一失

**Q: MVP 阶段的核心用户故事，你认为哪些必须优先实现？**

A: 用户能知道平台上有哪些产品正在制造。哪些维度的产品？比如说笔记类、效率类、营销类、视频类等等。然后也可以通过平台去筛选，比如说网页产品、APP产品等。

<br />

**Q: 你提到观众可以点「想用」「有趣」「期待」等反应，这些互动数据的用途是什么？**

A: 以上都要（包括：纯展示：在直播间实时显示反应数量，营造氛围（类似弹幕/emoji 雨）、影响推荐：反应数据作为推荐算法的信号，高「想用」的项目获得更多曝光、形成产品画像：汇总反应数据生成项目热度报告，帮主播了解观众对项目的兴趣分布）

**Q: 你提到用户可以按「笔记类、效率类、营销类」等维度浏览产品，这个产品目录/分类体系如何建立？**

A: 两者结合：主播手动设置基础分类，AI 辅助补充和优化标签

**Q: MVP 阶段你期望支持多少并发观众？这影响技术架构和第三方服务的选型：**

A: 小规模验证：单个直播间 50 人以内，平台总并发 500 人以内

**Q: 主播端的推流方式，你倾向于哪种实现？这决定了主播的使用门槛和开发复杂度：**

A: MVP 先用浏览器屏幕共享，后续再支持 OBS 推流

<br />

**Q: 你提到的「游戏化」具体想包含哪些元素？（上一轮未回答）**

A: 我指的更多是功能上的游戏化，或者说有游戏感。然后不跟传统的直播平台不一样。什么里的东西啊，就是这个。

<br />

**Q: 你提到「功能上的游戏化/有游戏感」，以下哪些具体设计方向最符合你的想法？**

A: 以上都要（包括：实时进度可视化：将 AI 编码过程包装成「建造」动画，代码行数变成建筑层数/进度条，项目完成时有庆祝特效、观众互动玩法：观众可以发送「加速」「Bug 炸弹」等虚拟道具影响直播间氛围，类似 Twitch Plays 的参与感、成就与收集系统：观众/主播通过观看、完成项目等行为解锁成就徽章，集齐特定系列可获得专属身份标识）

<br />

**Q: 关于「自动识别主播正在使用的 coding 工具」这个功能，你期望识别后做什么？**

A: 以上都要（包括：仅展示标签：在直播间显示「正在使用 Cursor / Copilot / Windsurf」等标识，方便观众了解、工具维度聚合：观众可以按工具筛选直播间（如「看所有用 Cursor 的主播」），形成工具生态、数据统计报告：汇总平台上各 AI 工具的使用占比、项目完成率等数据，生成趋势洞察）

<br />

**Q: 产品目录页（浏览正在被制造的产品）和直播间的关系是什么？**

A: 直播列表是首页核心：用户先看到正在直播的频道列表（类似 Twitch），产品信息作为直播间的附属描述

**Q: 一个产品（项目）的生命周期在平台上是怎样的？直播结束后内容如何处理？**

A: 直播 + 项目主页 + 最终产品链接：项目完成后可以发布上线链接，观众直接试用成品、

<br />

**Q: 用户账号体系和社交功能，MVP 阶段需要支持到什么程度？**

A: 最简：仅支持第三方登录（GitHub/Google），关注主播，收藏项目

<br />

**Q: MVP 阶段的核心页面和导航结构，你倾向于哪种设计？**

A: 极简三页：首页（直播列表）+ 直播间详情页 + 个人中心（关注/收藏）

**Q: 关于「智能隐藏 API Key / URL」的检测时机和处理方式，你期望怎么做？**

A: 延迟直播：所有直播流延迟 3-5 秒，服务端检测到敏感信息后在观众端裁剪/遮挡

<br />

**Q: 你提到的「实时进度可视化」「观众互动道具」「成就徽章」三大游戏化功能，MVP 阶段的优先级如何排？**

A: MVP 只做最轻量的「反应按钮（想用/有趣/期待）+ 进度条」，其余放到 V2

<br />

**Q: 平台的商业化模式，你目前有什么初步想法？这会影响功能设计（如虚拟道具是否付费、主播是否有收益）：**

A: 先不收费但预留商业化接口，MVP 重点跑通核心体验

<br />

**Q: 直播间的实时聊天功能，MVP 阶段需要支持到什么程度？**

A: 基础文字聊天 + 反应按钮都要，但不做弹幕/动画特效（包括：基础文字聊天：观众可以发送文字消息，支持简单的表情符号、结构化互动：除文字聊天外，支持反应按钮（想用/有趣/期待）和固定格式的提问（如「这行代码是什么意思？」）、MVP 不做聊天，只做反应按钮，聊天放到 V2）

<br />

**Q: 「智能隐藏 API Key / URL」功能中，延迟 3-5 秒的服务端检测方案需要处理哪些类型的敏感信息？请确认范围：**

A: 以上都要（包括：仅检测常见 API Key 格式（如 sk-xxx、AKIA-xxx 等固定前缀的密钥）、API Key + 环境变量 + .env 文件内容 + 数据库连接字符串、所有敏感信息：API Key、私有 URL（localhost/内网地址）、密码输入框、个人文件路径等）

<br />

**Q: 主播通过浏览器屏幕共享推流时，如果网络中断或浏览器崩溃，平台应该如何处理？**

A: 自动显示「主播暂时离开」占位画面，保持直播间不关闭，主播重连后自动恢复

**Q: 「自动识别主播正在使用的 coding 工具」在技术实现上，你倾向于哪种方式？（浏览器屏幕共享模式下）**

A: 基于屏幕画面的 AI 图像识别：通过截帧分析 UI 特征自动判断工具类型

<br />

**Q: 直播间的视觉风格和品牌调性，你期望是什么方向？你提到要「非常有个性」，具体指的是：**

A: 像素/复古游戏风：8-bit 风格 UI、像素动画、游戏化音效，强调趣味性和游戏感

<br />

**Q: MVP 的技术栈选型，后端和数据库你倾向于哪种组合？（前端已确定 Web 应用 + 第三方直播服务）**

A: Next.js 全栈（App Router + Server Actions）+ Supabase（PostgreSQL + 实时订阅 + Auth）

<br />

**Q: 项目主页（直播结束后的产品页面）需要展示哪些信息？这决定了数据模型设计：**

A: 以上都要（包括：基础信息：项目名称、描述、分类标签、使用的 AI 工具、最终产品链接、基础信息 + 数据统计：累计观看人数、反应数据（想用/有趣/期待）、开发耗时、基础信息 + 数据统计 + 回放精华：直播录像回放、AI 自动生成的项目开发摘要）

<br />

**Q: 关于直播间的「进度条」功能（MVP 游戏化的核心元素），进度如何确定？**

A: 主播手动设置阶段

<br />

**Q: MVP 的部署和上线策略，你倾向于哪种方式？**

A: 先用 Vercel + BaaS 快速上线 MVP，验证后再迁移到自建基础设施

<br />

**Q: 像素/复古游戏风的 UI 实现方案，你倾向于哪种？这影响开发工作量和视觉一致性：**

A: 混合风格：核心交互元素（反应按钮、进度条、成就徽章）用像素风，页面布局和文字用现代 UI，降低阅读负担

**Q: 在 Mux、Agora、LiveKit 这几个第三方直播服务中，你倾向于选择哪一个？它们各有不同的特点和定价模式：**

A: LiveKit：开源实时音视频框架，支持 WebRTC，延迟极低（< 1秒），适合需要服务端处理视频帧（如 AI 检测敏感信息）的场景

<br />

**Q: 你选择了延迟 3-5 秒的方案来做敏感信息检测。考虑到第三方直播服务的延迟特性，你能接受的最大端到端延迟（从主播屏幕到观众看到）是多少？**

A: 10-15 秒：安全优先，宁可延迟大一些也不能泄露敏感信息

<br />

**Q: 第三方直播服务的费用在 MVP 阶段（单直播间 50 人，平台总并发 500 人）你能接受的月度预算范围是？这直接影响服务选型：**

A: 尽量免费：优先选有免费额度的服务（如 LiveKit Cloud 免费层、Mux 每月免费分钟数）

<br />

**Q: 直播服务还需要支持录制功能（用于项目主页的回放精华），你期望录制方案如何实现？**

A: 使用直播服务自带的云端录制功能（如 Mux Asset、LiveKit Egress），自动存储到云存储

<br />
